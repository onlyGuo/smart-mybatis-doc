date: 2026-03-05
---

很多人第一次看到 SmartMybatis，会下意识把它归类到 “MyBatis-Plus 的替代品/竞品” 里。  
但把 `docs/guide/quick` 和 `docs/guide/dev` 通读一遍后，你会发现：**它们解决的痛点有重叠，设计路径却明显不一样**。

这篇文章不做“谁更好”的口水账，而是从真实开发链路出发：**接入成本、实体与映射、查询表达力（含联表）、分页排序、limit 分片查询、JSON 字段、初始化脚本、开发规范与团队协作、以及可回退/可控性**。最后我会给一张可快速决策的对比表。

---

## 先给结论（适合快速决策）

- **你想要“尽量贴近原生 MyBatis，同时把重复劳动交给框架做”**：SmartMybatis 更对味  
  - 文档里明确强调：**“只做增强不做封装”**（依旧建立在 MyBatis 官方依赖之上），可随时回退为纯 MyBatis 行为。
- **你想要“生态成熟、一套现成的工程化能力（代码生成、插件体系、IService/Wrapper 使用习惯）”**：MyBatis-Plus 更省心
- **你很在意：类型安全、IDE 友好、动态条件不写 if、Mapper 层可读性**：SmartMybatis 的 DSL 是“强主张”的路线
- **你希望开发期省掉大量 DDL 维护成本**：SmartMybatis 的 `auto-sync-db` 是很突出的卖点（但一定要控制环境与权限）

---

## 0. 对比前的“共同前提”（避免鸡同鸭讲）

- 二者都基于 MyBatis 生态（SQL 映射思想一致），都能在 Spring Boot 里用得很顺手。
- 真实项目里，你几乎一定会遇到：
  - 动态条件（可选筛选项多）
  - 分页/排序
  - 分片/分段拉取（limit/offset，流式搬运、导出、任务处理常见）
  - 联表查询（只取关联字段、或过滤语义很讲究）
  - “对象字段/列表字段”要么拆表、要么 JSON 存储
  - 命名策略与表前缀（历史库/多模块库常见）

下面的对比就围绕这些高频点展开。

---

## 1. 项目定位：增强的边界不同

### SmartMybatis：强调“增强 MyBatis，而不是封装 MyBatis”
在快速开始里写得很直白：

- SmartMybatis **不是重复造轮子**，而是“充分利用现有框架成熟特性，用更聪明的方式使用它们”
- **代码结构极简、性能强劲**，甚至强调“5 分钟读完全部源码并理解实现原理”（这是一种产品气质）
- `spring.mybatis.smart.enable=false` 可**完全禁用**，回退纯 MyBatis 行为（配置指南里明确写了）

> 这意味着：SmartMybatis 的核心目标不是“给你一套新的使用范式替换 MyBatis”，而是尽量让你仍在 MyBatis 世界里工作，只是更省事。

### MyBatis-Plus：更偏向“提供一套完整的效率体系”
MyBatis-Plus 的典型体验是：  
你会围绕它的 BaseMapper、Wrapper、Service 层范式、插件与工具链形成一套统一工作方式。生态成熟是它最大的优势，但也意味着“习惯迁移成本”和“框架范式绑定”更重。

---

## 2. 上手与配置：SmartMybatis 更“轻”，且强调“可回退”

### SmartMybatis 的接入（来自 quick guide）
- 依赖：`ink.icoding:spring-boot-starter-smart-mybatis:1.1.9`
- **前置条件**：必须先引入 **MyBatis 官方依赖**（文档注明：只增强不封装）
- 默认无需额外配置即可启用

配置项集中在：

```yaml
spring:
  mybatis:
    smart:
      enable: true
      naming-convention: underline_upper
      table-prefix: sm_
      auto-sync-db: true
```

几个点很“SmartMybatis”：
- `enable=false` 就彻底退回原生 MyBatis 行为（这对风险控制很关键）
- `naming-convention/table-prefix` 是一等公民
- `auto-sync-db` 明确面向开发期提效

### MyBatis-Plus 的接入（示意）
MyBatis-Plus 通常是 starter 方式接入，然后配置插件（分页、乐观锁等）。示意配置（仅用于对比写法）：

```yaml
mybatis-plus:
  configuration:
    map-underscore-to-camel-case: true
```

> 说明：本文涉及 MyBatis-Plus 的代码均为常见用法示例，目的是对比“写法与心智模型”；SmartMybatis 相关则严格以本仓库文档为准。

## 3. 实体与映射：SmartMybatis 更像“表结构描述”，并能驱动 schema 同步

### SmartMybatis：PO + @ID + 命名策略
SmartMybatis 文档（`entity.md`）对实体类有明确的基础规范：

1. **实体类必须继承 `PO`**
2. 必须有主键字段，并用 `@ID` 标识

并支持：
- 默认命名策略：类名/字段名转下划线大写（可配置 `naming-convention`）
- `@TableName` 自定义表名
- `@TableField` 自定义列名、长度、列类型、描述等

示例（SmartMybatis 风格，来自文档结构）：

```java
@TableName("users")
public class User extends PO {
    @ID
    private Long id;

    @TableField(length = 50, description = "用户名")
    private String username;

    private String email;
}
```

主键生成策略也有明确枚举（`primary-generator.md`）：

```java
public class Example extends PO {
    @ID(generateType = PrimaryGenerateType.SNOWFLAKE)
    private Long id;
}
```

### MyBatis-Plus：TableName/TableId/TableField（常见写法示例）

```java
@TableName("users")
public class UserEntity {
    @TableId(type = IdType.ASSIGN_ID) // 常见：雪花/分布式
    private Long id;

    @TableField("username")
    private String username;

    private String email;
}
```

两者“表面”都在用注解映射，但 SmartMybatis 在文档里会更强调：
- 命名策略（`naming-convention`）贯穿一切
- 注解里提供的字段结构信息，可以配合 `auto-sync-db` 直接减少 DDL 维护工作量

## 4. 动态条件与 DSL：SmartMybatis 强推“ifAnd + 方法引用”来打掉 if 地狱

这是 SmartMybatis 文档里反复强调的核心体验：

### SmartMybatis：Where DSL（文档示例风格）

```java
public List<User> searchUsers(String username, String email) {
    return userMapper.select(
        Where.where()
            .ifAnd(User::getUsername).like(username)
            .ifAnd(User::getEmail).eq(email)
    );
}
```

关键语义（文档明确说明）：
- `ifAnd/ifOr` 遇到 `null` 或空字符串会跳过条件
- 若要生成 `IS NULL/IS NOT NULL`，请用标准 `and/or`
- `LIKE` 会做“智能适配”

### MyBatis-Plus：LambdaQueryWrapper（常见写法示例）

```java
public List<UserEntity> searchUsers(String username, String email) {
    return userMapper.selectList(
        Wrappers.<UserEntity>lambdaQuery()
            .like(username != null && !username.isBlank(), UserEntity::getUsername, username)
            .eq(email != null && !email.isBlank(), UserEntity::getEmail, email)
    );
}
```

对比一下“写法的心理压力”：
- MP 的动态条件通常靠 `condition` 参数解决，但你仍要在每个条件旁边写判断；
- SmartMybatis 则把“判空跳过”做成了 DSL 的默认行为（`ifAnd`），更像是在写自然语言。

## 5. 联表查询：SmartMybatis 把 JOIN 收敛进 Where，并支持“字段填充”

联表是很多项目最后不得不回到 XML/手写 SQL 的地方。SmartMybatis 的文档里（`join-query-*.md`）给了一套非常“面向业务填充”的写法：**LEFT JOIN + 把关联表字段填充回主实体的临时字段**。

### 5.1 SmartMybatis：最小可用的 leftJoin（来自 `join-query-basic.md`）

实体示意（文档要点：`exist=false` 表示非表字段，仅用于承载关联结果；并配置 link/linkField）：

```java
@TableName("order")
public class Order {
    @ID
    private Long id;

    private Long userId;

    @TableField(exist = false, link = User.class, linkField = "name")
    private String userName;
}
```

查询并填充：

```java
List<Order> orders = orderMapper.select(
    Where.where().leftJoin(
        User.class,
        "u",
        Where.where(User::getId).eq(Order::getUserId),
        Order::getUserName
    )
);
```

文档给的 SQL 直觉（仅用于理解）：

```sql
SELECT o.*, u.name AS user_name
FROM `order` o
LEFT JOIN `user` u ON u.id = o.user_id;
```

### 5.2 SmartMybatis：JOIN ON 过滤 vs WHERE 过滤（来自 `join-query-filter.md`）

同样是“只看用户姓名包含张的订单”，两种写法语义不同：

- **WHERE 过滤**：会把主表订单也过滤掉

```java
Where joinOn = Where.where(User::getId).eq(Order::getUserId);

List<Order> orders = orderMapper.select(Where.where()
    .leftJoin(User.class, "u", joinOn, Order::getUserName)
    .like(Order::getUserName, "%张%")
);
```

- **JOIN ON 过滤**：不影响主表行数，只影响关联字段是否能填充（填充不到则为 null）

```java
Where joinOn = Where.where(User::getId)
    .eq(Order::getUserId)
    .and(User::getName).like("%张%");

List<Order> orders = orderMapper.select(Where.where()
    .leftJoin(User.class, "u", joinOn, Order::getUserName)
);
```

这段文档非常“工程化”：它把很多人线上踩坑的语义差异写成了明确对照。

### 5.3 SmartMybatis：多表 leftJoin 串联（来自 `join-query-multi.md`）

```java
Where joinUser = Where.where(User::getId).eq(Order::getUserId);
Where joinItem = Where.where(OrderItem::getOrderId).eq(Order::getId);
Where joinProduct = Where.where(Product::getId).eq(OrderItem::getProductId);

List<Order> orders = orderMapper.select(Where.where()
    .leftJoin(User.class, "u", joinUser, Order::getUserName)
    .leftJoin(OrderItem.class, "oi", joinItem)
    .leftJoin(Product.class, "p", joinProduct, Order::getProductName)
);
```

### 5.4 MyBatis-Plus：联表通常的落点（常见实践说明）
MyBatis-Plus 本身更擅长单表 CRUD 与 Wrapper 条件构造。联表常见有三条路：

1. **写 XML/注解 SQL**（最常见、最可控）
2. **使用 MPJ 等联表扩展**（引入额外库，换来 Wrapper 风格联表）
3. 退回 MyBatis 原生的 Provider/XML 能力

示意（XML/注解 SQL 的常见写法，仅用于对比）：

```java
@Mapper
public interface OrderMapper {
    @Select("SELECT o.*, u.name AS user_name FROM `order` o LEFT JOIN `user` u ON u.id=o.user_id")
    List<OrderDTO> listWithUserName();
}
```

> 对比点不在“MP能不能联表”，而在于：SmartMybatis 把联表这件事纳入了同一套 Where DSL（并提供字段填充语义）；MP 的联表则更多依赖手写 SQL 或额外生态。

## 6. 分页与排序：SmartMybatis 内置直观 API，但也不强绑定某个分页插件

### SmartMybatis：selectPage + orderBy（来自 `query-page.md` / `query-sort.md`）

```java
public PageResult<Student> searchStudent(
        String name, Integer minAge, Integer maxAge, Sex sex, Page page) {

    Where where = Where.where()
        .ifAnd(Student::getName).like(name)
        .ifAnd(Student::getAge).greaterThanOrEquals(minAge)
        .ifAnd(Student::getAge).lessThanOrEquals(maxAge)
        .ifAnd(Student::getSex).equalsFor(sex)
        .orderBy(Student::getAge).desc();

    return studentMapper.selectPage(where, page);
}
```

### MyBatis-Plus：Page + selectPage（常见写法示例）

```java
public IPage<UserEntity> pageUsers(int page, int size, String username) {
    Page<UserEntity> p = new Page<>(page, size);

    return userMapper.selectPage(
        p,
        Wrappers.<UserEntity>lambdaQuery()
            .like(username != null && !username.isBlank(), UserEntity::getUsername, username)
            .orderByDesc(UserEntity::getId)
    );
}
```

## 7. limit 分片/分段查询：SmartMybatis 把 limit 作为 Where 的能力点

很多时候我们不需要“分页总数”（比如批处理、导出、异步任务拉取），只是想分段取数据。

### SmartMybatis：limit(10) / limit(offset, size)（来自 `query-limit.md`）

```java
Where where = Where.where()
    .ifAnd(Student::getSex).eq(sex)
    .orderBy(Student::getId).asc()
    .limit(20, 10); // offset=20, size=10

List<Student> data = studentMapper.select(where);
```

文档强调：`limit` 加入的位置不讲究先后（语义上会生成对应 LIMIT 子句）。

### MyBatis-Plus：分页/滚动拉取的常见写法（示意）
MP 通常用 Page 做“无 total”的滚动拉取，或者用 `last("limit ...")` 拼接：

```java
List<UserEntity> data = userMapper.selectList(
    Wrappers.<UserEntity>lambdaQuery()
        .gt(UserEntity::getId, lastId)
        .orderByAsc(UserEntity::getId)
        .last("LIMIT 10")
);
```

> 这里也体现两者风格：SmartMybatis 把 limit 覆盖到 DSL 里；MP 更常通过分页对象或 `last(...)` 达成。

## 8. JSON 字段：SmartMybatis 用注解把“对象字段 JSON 存储”变成开箱即用

### SmartMybatis：@TableField(json = true)（来自 `json-field.md`）

```java
public class UserProfile extends PO {
    @ID
    private Long id;

    private String userName;

    @TableField(json = true)
    private Address address;
}

public class Address {
    private String street;
    private String city;
    private String zipCode;
}
```

### MyBatis-Plus：TypeHandler（常见写法示例）

```java
@TableName("user_profile")
public class UserProfileEntity {
    @TableId
    private Long id;

    private String userName;

    @TableField(typeHandler = JacksonTypeHandler.class)
    private Address address;
}
```

## 9. 初始化 SQL 脚本：SmartMybatis 可以把 init.sql “绑在表上”

这点在很多业务初始化、演示环境、或需要靠 SQL 精修表结构的时候很实用。

### SmartMybatis：@TableName(init = "init.sql")（来自 `init-sql.md`）

```java
@TableName(name = "example_table", init = "init.sql")
public class Example extends PO {
    @ID
    private Long id;
}
```

文档要点：
- `init.sql` 路径是 classpath 下的资源
- 只有当表不存在或表内无数据时才执行；表里已存在任意数据则不会执行
- 当开启自动同步表结构时，init 脚本会在结构同步后执行
- init 脚本甚至可以在“表数据为空”时用于覆写自动同步出来的结构，做精修

### MyBatis-Plus：通常交给迁移工具/应用初始化逻辑
MP 自身不提供“把 init.sql 绑定到实体”这类能力。常见做法是：
- Flyway/Liquibase 迁移脚本
- Spring Boot 的 `schema.sql`/`data.sql`
- 应用启动时业务初始化（手写）

## 10. 开发期自动同步表结构：SmartMybatis 的“强卖点”，也需要强约束

### SmartMybatis：auto-sync-db（来自 `auto-sync-db.md` / 配置指南）

```yaml
spring:
  mybatis:
    smart:
      auto-sync-db: true
```

文档明确提示：
- 启动时检查实体与表结构差异，尝试同步新增字段
- **生产环境谨慎使用**
- 配置指南补充：目前仅支持新增/修改字段，不删除字段，以降低风险

### MyBatis-Plus：更多依赖迁移工具链
MP 更典型的路线是：Flyway/Liquibase + 代码生成/手工维护 DDL。

## 11. 工程实践与团队协作：SmartMybatis 文档对“怎么写更可维护”有强主张

来自 `guidelines.md` 的两条建议尤其值得抄进团队规范：

1. **不建议在 Service 层构建复杂查询条件**  
   把条件封装在 Mapper 层 default 方法或专门的查询类中。
2. **不建议直接用 PO 当入参/出参**  
   建议用 DTO/VO 做边界层数据传递。

## 12. 什么时候选谁？更细一点的选型建议

### 更推荐 SmartMybatis 的情况
- 你希望**保留 MyBatis 的控制力**（必要时随时写 XML/注解 SQL）
- 你很看重 **可回退**（`enable=false`）
- 复杂筛选条件多，想用 DSL 把 if/拼接/手写条件统一收敛
- 联表查询希望尽量“DSL 化”，并且需要字段填充+语义明确（ON vs WHERE）
- 批量任务/导出这种场景多，希望 limit 分段查询表达简单
- 你希望开发期减少 DDL 维护，并有能力把风险控制在开发环境

### 更推荐 MyBatis-Plus 的情况
- 你要的是成熟生态与统一范式：Wrapper + Service 抽象 + 插件体系
- 团队成员对 MP 生态更熟，能快速落地
- 你希望“按社区主流写法”招人、交接、维护更顺滑

## 13. 快速对比表（建议收藏）

| 维度 | SmartMybatis（基于本文档） | MyBatis-Plus（常见实践） | 你该怎么选 |
|---|---|---|---|
| 定位 | 只增强不封装，强调保留 MyBatis 控制力 | 提供完整效率体系与统一范式 | 想贴近 MyBatis：Smart；想一套现成体系：MP |
| 接入与回退 | `spring.mybatis.smart.enable=false` 可完全禁用回退 | 一旦使用其范式/生态，迁移回退成本更高 | 对“可控性/回退”敏感：Smart |
| 命名策略/表前缀 | `naming-convention`、`table-prefix` 是核心配置 | 通常依赖 MyBatis/MP 配置与注解分散控制 | 历史库/多模块前缀强：Smart 更顺 |
| 动态条件 | `Where.where().ifAnd(...).xxx()` 判空跳过成为默认行为 | Wrapper 的 `condition` 参数常用，但仍需显式判断 | 条件很多且追求可读性：Smart |
| 联表查询 | `leftJoin` + 字段填充，且区分 ON/WHERE 过滤语义 | 多数落到 XML/手写 SQL 或引入联表扩展 | 想把联表纳入同一 DSL：Smart |
| 分页/排序 | `selectPage(where, page)` + `orderBy().asc/desc` | Page + 插件体系成熟 | 生态成熟优先：MP；想轻量直观：Smart |
| limit 分片查询 | `Where.limit(size)` / `limit(offset, size)` 是 DSL 一部分 | 常用 `Page` 或 `last("limit")` | 批处理/导出多：Smart 更顺 |
| JSON 字段 | `@TableField(json = true)` 开箱即用，默认 Jackson，列类型 TEXT | 多用 TypeHandler（如 JacksonTypeHandler） | 想最省配置：Smart |
| 初始化脚本 | `@TableName(init = "init.sql")` 绑定表初始化脚本 | 常交给 Flyway/Liquibase 或 Boot 的 data.sql | 想“表级 init”开箱即用：Smart |
| schema 变更 | `auto-sync-db` 开发期可自动同步新增字段（有风险提示） | 通常用迁移工具链 | 想开发期极致快：Smart；严格流程：MP/迁移 |
| 工程规范 | 文档明确建议：查询条件收口 Mapper、PO 不做入出参 | 依赖团队/社区约定 | 希望框架提供“官方写法”：Smart |

## 14. 一句话总结

- **SmartMybatis** 更像：*“我仍然是 MyBatis，但我把你每天重复写的那部分变得更聪明、更类型安全、更可维护。”*
- **MyBatis-Plus** 更像：*“我给你一整套高效率的 MyBatis 使用体系，从 CRUD 到生态工具链都给齐。”*

如果你正在做技术选型，我建议你先问自己两个问题：
1. 你希望团队的默认心智模型是 **MyBatis** 还是 **MyBatis-Plus**？
2. 你更想要的是 **“轻量增强 + 可控回退”**，还是 **“成熟体系 + 统一范式”**？

答案通常就很明确了。
